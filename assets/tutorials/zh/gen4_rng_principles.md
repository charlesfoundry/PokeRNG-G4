# Gen4乱数原理

第四世代乱数可以理解为：

```text
机器时间 + Delay 生成初始 seed
初始 seed 后面的固定随机序列决定宝可梦结果
玩家操作、NPC、电话、喋喋不休等行为会推进这条序列
```

所以实机乱数的目标是：

1. 命中正确的 `seed`。
2. 推进到正确的 `frame / advance`。
3. 在正确的位置触发宝可梦生成。

## 核心概念

| 名称 | 通俗解释 | 实机意义 |
| --- | --- | --- |
| `seed` | 随机数序列的起点 | 命中同一个 seed，后面的随机序列才会一致 |
| `Delay` | 游戏启动到读取 seed 时经过的硬件偏移 | 决定能否命中目标 seed |
| `second` | 生成 seed 时使用的秒数 | DS 时间只能设置到分钟，秒数靠玩家等待 |
| `frame / advance` | RNG 被消耗了多少次 | 决定使用 seed 后面第几个结果 |
| `PID` | 宝可梦的隐藏人格值 | 决定性格、性别、特性槽、闪光等 |
| `IV` | 个体值 | 通常由 PID 后面的 RNG 结果决定 |
| `TID/SID` | 表/里训练家 ID | 和 PID 一起决定是否闪光 |

## seed 是怎么由时间决定的

Gen4 的初始 seed 由 DS 机器时间和 Delay 组合出来。

可以简化理解为：

```text
AB = 月 * 日 + 分 + 秒
CD = 小时

seed = (AB << 24) | (CD << 16)
seed = seed + Delay + 年份修正
```

也就是说：

```text
年月日时分秒  决定 seed 的高位
Delay         决定 seed 的低位
年份          会修正 Delay 显示值
```

这也是为什么同一个 seed 可以反推出很多日期时间：不同日期、分钟、秒数可能算出同样的高位组合。

> 对玩家来说，重点不是某个日期本身，而是这个日期时间和 Delay 能不能生成目标 seed。

## Delay 是什么

`Delay` 可以理解为：

```text
从游戏启动到游戏读取初始 seed 之间，
经过了多少个硬件计时单位。
```

在通常的实机操作中，可以把它近似理解为：

```text
从 DS 系统菜单按 A 启动游戏，
到选择存档画面按 A 进入存档之间，
这段操作时间对应出的启动偏差。
```

它不是普通的游戏帧，也不是要等待的秒数。

例如目标显示：

```text
Seed: 00130AE9
Delay: 2782
Second: 57
```

意思是：

```text
目标是在某个时间的第57秒，
以 Delay 2782 这个启动偏差进入游戏，
从而得到 seed 00130AE9。
```

定时器的意义，就是帮助玩家稳定做到这件事。

## 为什么需要校准

玩家不可能第一次就精确命中目标 Delay。

实际流程通常是：

```text
选择目标 seed / Delay / second
用定时器进入游戏
用电话、硬币或实际捕获结果确认命中
反推出实际 Delay
应用校准
再次尝试
```

例如：

```text
目标 Delay: 2782
实际命中:   2772
```

App 会根据这个偏差更新校准 Delay，让下一次定时器更贴近你的实际手感。

## 年份和 Delay 奇偶

Gen4 的 seed 公式包含年份修正：

```text
Delay + year - 2000
```

因此年份会影响可命中的 Delay 奇偶。

如果出现：

```text
目标 Delay 是偶数
实际命中总是奇数
```

或者相反，通常不需要改 App 的筛选条件，只需要把 DS 主机年份加 1 或减 1，即可切换奇偶。

## seed 后面发生什么

游戏拿到初始 seed 后，会按固定公式不断生成下一个随机数：

```text
seed = seed * 0x41C64E6D + 0x6073
```

每生成一次随机数，就可以看作 RNG 往后推进了一次。

所以：

```text
同一个 seed + 同一个推进次数 = 同一个结果
```

宝可梦的遇敌槽位、等级、PID、性格、性别、特性槽、个体值、道具和部分形态，都是从后续随机数里读出来的。

## 电话、硬币、喋喋不休

### HGSS 电话

HGSS 电话可以用来确认 seed，也会推进 RNG。

每打一次博士或杂技艺人电话，会得到 `E/K/P` 之一。App 可以根据玩家输入的电话序列反查：

```text
哪个 seed 会产生这串 E/K/P？
这串电话开始前，当前已经处于第几帧？
```

### DPPt 硬币

DPPt 可以用硬币结果观察 seed 后面的序列。它主要用于确认 seed。

### 喋喋不休

喋喋不休主要用于推进帧。

每听到一次自制录音：

```text
RNG 消耗 1 次
玩家前进 1 帧
音高由这次随机数决定
```

## Method 1 / J / K

Gen4 不是所有宝可梦都用同一套读取方式。

| Method | 常见用途 | 说明 |
| --- | --- | --- |
| Method 1 | 御三家、礼物、部分特殊生成 | 读取流程比较直接 |
| Method J | DP/Pt 定点、野生等 | 会处理同步、性格筛选、遇敌槽等 |
| Method K | HGSS 定点、野生等 | HGSS 对应的读取方式 |

它们共用同一个初始 seed 和 LCRNG，但读取随机数的顺序不同。

所以同一个 seed、同一帧，用不同 Method 算出来的结果可能完全不同。

## 野生遇敌为什么复杂

野生遇敌不只是生成 PID 和 IV。它通常还要计算：

- 是否遇敌
- 遇敌槽位
- 等级
- 物种
- 性格和 PID
- 个体值
- 道具
- 部分特殊形态

队首特性也会影响流程：

| 队首特性 | 可能影响 |
| --- | --- |
| 同步 | 性格 / PID 生成 |
| 迷人之躯 | PID 池和性别结果 |
| 静电 / 磁力 | 特定属性物种 |
| 压迫感 | 等级 |
| 复眼 | 道具 |

所以检索时，App 里的队首设置必须和实机一致。

## ID 乱数

ID 乱数也是由时间 seed 决定的，但生成 TID/SID 时不是直接用普通 LCRNG，而是从 seed 启动 MT。

简化流程：

```text
时间 + Delay -> seed
seed -> MT -> TID/SID
```

TID/SID 重要是因为闪光判断使用：

```text
TID ^ SID ^ PID高位 ^ PID低位
```

如果能控制 TID/SID，就能让某些优秀 PID 变成闪光。

## 孵蛋为什么分两段

Gen4 孵蛋要分成：

```text
生成蛋
领取蛋
```

生成蛋决定：

- PID
- 性格
- 性别
- 特性槽
- 是否闪光

领取蛋决定：

- 个体值
- 父母遗传

所以已经生成、但还没有领取的蛋，可以继续重启游戏刷领取阶段的个体值。

## 最容易出错的地方

### 1. 把 Delay 当成 frame

`Delay` 是启动阶段偏差。
`frame / advance` 是 seed 之后 RNG 被消耗的次数。

它们不是同一个东西。

### 2. seed 对了，但 frame 不对

常见原因：

- NPC 转向或移动
- 游走宝可梦初始消耗
- 电话次数和 App 点击次数不一致
- 喋喋不休次数数错
- 触发遇敌方式理解错

### 3. 搜索条件和实机状态不一致

例如：

- App 没选同步，但游戏队首是同步
- 时间段选择错误
- 卡带插入状态错误
- 地点或遇敌方式选错
- 等级范围或槽位理解错误

## 一句话总结

```text
Gen4 乱数不是赌随机，
而是用 DS 时间和 Delay 命中一个 seed，
再用可控操作把 RNG 推进到目标位置。
```
